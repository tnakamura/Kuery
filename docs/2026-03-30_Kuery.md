# 軽量ORM「Kuery」を作りました: ADO.NETで気持ちよく使える、ちょうどいいデータアクセス

[Kuery](https://github.com/tnakamura/Kuery) を開発しました！

- GitHub: https://github.com/tnakamura/Kuery
- NuGet: https://www.nuget.org/packages/Kuery

「Entity Framework Core は少し重い」「Dapper は好きだけど、もう少しクエリビルダーの気持ちよさがほしい」。
その間を埋めるために、ADO.NET 上で使えるシンプルな ORM / クエリライブラリとして Kuery を実装しています。

さらに言うと、私は `sqlite-net` の設計がとても好きです。
この「素朴でわかりやすく、必要十分に速く、小さくまとまっている」感覚を、.NET の普段使いでもっと活かしたい。
それが Kuery の出発点です。

---

## なぜ Kuery なのか

データアクセスの選択肢は多いですが、実際の開発ではこういう悩みがよくあります。

- **EF Core**: 強力で表現力も高い。ただ、用途によっては「機能が大きすぎる」と感じることがある  
- **Dapper**: とても良い。速いし素直。ただ、クエリ組み立てや共通化をもう一歩だけ手元で整えたいことがある  
- **生の ADO.NET**: 完全に制御できる反面、毎回のボイラープレートがつらい  

Kuery はこの間にある「ちょうどいい層」を狙っています。

- ADO.NET の `IDbConnection` をそのまま拡張
- 属性ベースのシンプルなマッピング
- `Table<T>()` で直感的な CRUD
- `Query<T>()` で LINQ から SQL を生成
- 必要なら手書き SQL にすぐ戻れる

つまり、**素の ADO.NET の現実感を保ったまま、開発速度だけ上げる**ためのライブラリです。

---

## 使い方はシンプル

例えば Todo テーブルを扱うとき、モデルはこう書けます。

```csharp
[Table("todo")]
public class Todo
{
    [PrimaryKey]
    [Column("id")]
    [AutoIncrement]
    public int Id { get; set; }

    [NotNull]
    [Column("name")]
    public string Name { get; set; } = "";

    [Column("description")]
    public string? Description { get; set; }
}
```

挿入は extension method を呼ぶだけです。

```csharp
using var connection = new SqliteConnection("Data Source=app.db");
connection.Open();

var todo = new Todo { Name = "Kueryを試す", Description = "まずは Insert から" };
connection.Insert(todo);
```

検索も自然です。

```csharp
var rows = connection.Table<Todo>()
    .Where(x => x.Name == "Kueryを試す")
    .OrderBy(x => x.Id)
    .ToList();
```

もちろん async も使えます。

```csharp
await connection.InsertAsync(todo, cancellationToken: ct);

var list = await connection.Table<Todo>()
    .Where(x => x.Id > 0)
    .ToListAsync(ct);
```

## `Query<T>()` で、LINQ から SQL を組み立てる

Kuery のもうひとつの軸が `Query<T>()` です。
LINQ で書いた式を SQL に変換して実行できます。

```csharp
var products = await connection.Query<Product>()
    .Where(p => p.Code == "0001")
    .OrderBy(p => p.Code)
    .ToListAsync();
```

さらに更新・削除も、条件を明示した形で実行できます。

```csharp
var deleted = await connection.Query<Product>()
    .Where(p => p.Id == id)
    .ExecuteDeleteAsync();

var updated = await connection.Query<Product>()
    .Where(p => p.Code == "0001")
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(p => p.Name, p => p.Name + " (updated)"));
```

「全部を抽象化しきる」のではなく、**現場でよく使う LINQ をしっかり SQL に落とす**ことに集中しています。
必要なら `Query<T>(sql, param)` の手書き SQL へ即座に切り替えられるので、運用上の安心感もあります。

## 設計の思想: 重厚さではなく、制御しやすさ

Kuery は、次のようなバランスを最重視しています。

- **軽いこと**: 学習コストと導入コストを下げる
- **わかりやすいこと**: SQL が頭に入っている人ほど扱いやすい
- **逃げ道があること**: 困ったら手書き SQL に戻れる
- **ADO.NET の上に立つこと**: 接続・トランザクション・運用知識をそのまま活かせる

この思想は `sqlite-net` の「過剰に隠蔽しない」アプローチから強く影響を受けています。
アプリケーション開発では、万能な抽象化よりも、**壊れたときに直せる設計**が最終的な強さになります。

## どんな人に使ってほしいか

Kuery は特にこんなチームに刺さるはずです。

- EF Core だと少し重いと感じている
- Dapper のよさは理解しているが、もう少しモデル寄りに書きたい
- SQL は読み書きできるので、ブラックボックス化は不要
- ADO.NET ベースで堅実に運用したい

小さな業務アプリ、API バックエンド、社内ツール、PoC から本番まで。
「速度と保守性を両立したい」現場で、Kuery はかなり使いやすい選択肢だと思っています。
