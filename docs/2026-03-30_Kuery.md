# ADO.NETの現場を救うために、軽量ORM「Kuery」を作った

[Kuery](https://github.com/tnakamura/Kuery) を開発しています。  
NuGet: https://www.nuget.org/packages/Kuery

最初に結論を書きます。  
**EF Core が重いと感じるプロジェクト、でも Dapper の素の SQL だけでは少し足りないプロジェクト、その間に Kuery は刺さります。**

私はこの「間」がずっと気になっていました。

- EF Core はすごい。機能も厚い。だけど、アプリによっては重厚すぎる
- Dapper は本当に良い。速いし、素直だし、好き
- ただ、Dapper を使っていて「ここ、もう少しクエリビルダー欲しいな」が何度も出る

そしてもう一つ。  
私は `sqlite-net` の設計思想がかなり好きです。  
「抽象化はする。でも隠しすぎない。小さい。速い。分かる。」  
この感覚を ADO.NET の日常に持ち込みたくて、Kuery を実装しました。

---

## 目指したものは、万能ORMではなく「強い実用品」

Kuery の狙いはシンプルです。

- ADO.NET の `IDbConnection` をそのまま使える
- 属性ベースでテーブル/カラムを素直にマッピングできる
- CRUD を短く書ける
- `Query<T>()` で LINQ から SQL を組み立てられる
- そして必要なら即座に手書き SQL に戻れる

つまり、**現場の制御感を絶対に失わないまま、書く量だけを減らす**ためのライブラリです。

重い抽象化で「便利なはずなのに怖い」を作るのではなく、  
軽い抽象化で「読める・追える・直せる」を作る。  
これが Kuery の芯です。

---

## 使うときの温度感

モデルはこのぐらい素朴に書けます。

```csharp
[Table("todo")]
public class Todo
{
    [PrimaryKey]
    [Column("id")]
    [AutoIncrement]
    public int Id { get; set; }

    [NotNull]
    [Column("name")]
    public string Name { get; set; } = "";

    [Column("description")]
    public string? Description { get; set; }
}
```

Insert は extension method を呼ぶだけ。

```csharp
using var connection = new SqliteConnection("Data Source=app.db");
connection.Open();

var todo = new Todo { Name = "Kueryを試す", Description = "最初の1行" };
connection.Insert(todo);
```

検索も、欲しい形でそのまま書けます。

```csharp
var rows = connection.Table<Todo>()
    .Where(x => x.Name == "Kueryを試す")
    .OrderBy(x => x.Id)
    .ToList();
```

async も同じ温度で使えます。

```csharp
await connection.InsertAsync(todo, cancellationToken: ct);

var list = await connection.Table<Todo>()
    .Where(x => x.Id > 0)
    .ToListAsync(ct);
```

## `Query<T>()`: Dapperの現実感に、クエリビルダーの気持ちよさを足す

Kuery を語るうえで外せないのが `Query<T>()` です。

```csharp
var products = await connection.Query<Product>()
    .Where(p => p.Code == "0001")
    .OrderBy(p => p.Code)
    .ToListAsync();
```

更新と削除も、条件を明示しながら一貫して書けます。

```csharp
var deleted = await connection.Query<Product>()
    .Where(p => p.Id == id)
    .ExecuteDeleteAsync();

var updated = await connection.Query<Product>()
    .Where(p => p.Code == "0001")
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(p => p.Name, p => p.Name + " (updated)"));
```

ここで大事なのは「ORM に全部任せる」ではないことです。  
**必要なところは LINQ で素早く、難しいところは SQL を直接書く。**  
その行き来が自然にできるのが Kuery の価値です。

## なぜこの方向なのか

現場で本当に効くのは、だいたい次の4つです。

1. 読みやすい  
2. 追跡しやすい  
3. 問題が起きたときに戻せる  
4. パフォーマンスの勘所を失わない

`sqlite-net` が好きな理由もここにあります。  
過剰に隠蔽しないから、壊れたときに自分で立て直せる。

Kuery も同じです。  
「魔法」ではなく「手になじむ道具」を目指しています。

---

## 使ってほしい人

- EF Core が合わないわけではないけど、今の案件には重い
- Dapper の思想は好き。だけどモデルとの往復をもう少し楽にしたい
- SQL を読める/書けるので、抽象化の中に閉じ込められたくない
- ADO.NET ベースで堅実に運用したい

こういうチームには、かなり気持ちよくハマるはずです。
